## Executive Summary

This contract defines the incremental parsing work for adze v0.9.0, establishing editor-class performance through intelligent subtree reuse and GLR-aware incremental parsing.

**Goal**: Achieve **≤30% of full parse time** for typical single-line edits while maintaining **100% correctness** and preserving GLR ambiguity information.

**Current State** (as of 2025-11-20):
- ⚠️  Incremental parsing partially implemented (feature-gated)
- ⚠️  `Tree::edit()` exists but incomplete
- ⚠️  No incremental parsing validation tests
- ⚠️  GLR-specific incremental semantics undefined
- ✅ Performance baseline established for full parsing

**Timeline**: 9 weeks (Weeks 7-15 of roadmap, ending Q1 2026)

**Requirement**: For each Tree-sitter incremental API concept, provide a adze equivalent with identical semantics

**Tree-sitter API Surface**:
```c

typedef struct {
  uint32_t start_byte;
  uint32_t old_end_byte;
  uint32_t new_end_byte;
  TSPoint start_point;
  TSPoint old_end_point;
  TSPoint new_end_point;
} TSInputEdit;

void ts_tree_edit(TSTree *self, const TSInputEdit *edit);

```

**Rust-sitter Equivalent**:
```rust
// runtime2/src/edit.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Edit {
    pub start_byte: usize,
    pub old_end_byte: usize,
    pub new_end_byte: usize,
    pub start_position: Point,
    pub old_end_position: Point,
    pub new_end_position: Point,
}

impl Edit {

        &mut self,
        input: &[u8],
        old_tree: Option<&Tree>,
    ) -> Result<Tree, ParseError>;
}
```

### AC-INCR2: Edit Semantics & Correctness

**Requirement**: Incremental parse must produce identical results to full parse, regardless of edit history

**Correctness Invariant**:
```rust
// Golden test property
for any (original_text, edits_sequence) {
    let tree_incremental = apply_edits_incrementally(original_text, edits_sequence);
    let tree_full = parse_full(apply_edits_to_text(original_text, edits_sequence));

    assert_tree_equal(tree_incremental, tree_full);
}
```

**Edit Operations**:
- **Insert**: Add text at position (e.g., typing character)
- **Delete**: Remove text range (e.g., backspace, selection delete)
- **Replace**: Delete + insert (e.g., paste over selection)

**Dirty Region Tracking**:
- Mark nodes affected by edit as "dirty"
- Compute LCA (Lowest Common Ancestor) of dirty region
- Extend to safe reparse boundaries (statement/expression boundaries)

**Success Criteria**:
- [ ] Property test: incremental == full parse (1000+ random edit sequences)
- [ ] Correctness test suite: 50+ hand-crafted edit scenarios
- [ ] No "incremental drift" (repeated edits don't accumulate errors)
- [ ] GLR ambiguity preserved across incremental parses

### AC-INCR4: GLR-Aware Incremental Parsing

**Requirement**: Incremental parsing must correctly handle GLR ambiguity and fork/merge operations

**GLR-Specific Challenges**:
1. **Ambiguity Preservation**: If old tree had ambiguity, incremental parse must preserve or update it
2. **Fork Boundaries**: Reparse window must not break GLR fork points
3. **Forest Updates**: Update parse forest, not just single tree

**Design**:
```rust
// runtime2/src/incremental_glr.rs
pub struct IncrementalGLRParser {
    engine: GLREngine,
    forest_cache: ForestCache,
}

impl IncrementalGLRParser {
    /// Compute reparse window respecting GLR fork boundaries
    fn compute_reparse_window(
        &self,
        edit: &Edit,
        old_tree: &Tree,
    ) -> ReparseWindow;

    /// Incrementally reparse, preserving ambiguity
    fn reparse_incremental(
        &mut self,
        input: &[u8],
        old_tree: &Tree,
        window: ReparseWindow,
    ) -> Result<Tree, ParseError>;

    /// Stitch new subtree back into old tree
    fn stitch_subtree(
        &self,
        old_tree: Tree,
        new_subtree: Tree,
        window: ReparseWindow,
    ) -> Tree;
}
```

**Reparse Window Calculation**:
1. Find LCA of edited region
2. Extend to statement/expression boundaries
3. Add lookahead padding (5-10 tokens before/after)
4. Check for GLR fork points, extend if necessary

**Success Criteria**:
- [ ] Ambiguous grammars supported incrementally
- [ ] Fork points preserved or correctly updated
- [ ] Parse forest incremental update works
- [ ] No spurious ambiguity introduced by incremental parsing

**BDD Scenarios**: See Section V, Scenario 4

## IV. Performance Budget

### Latency Budget (Editor Responsiveness)

**Target Latencies**:
- Single-character insert: ≤10ms (95th percentile)
- Single-line edit: ≤30ms (95th percentile)
- 10-line paste: ≤100ms (95th percentile)
- Large structural change (100+ lines): ≤500ms or fallback to full

**Breakdown**:
```
Total Budget (30ms for 1-line edit):
  - Edit application: 0.5ms
  - Dirty region calculation: 2ms
  - Reparse window computation: 3ms
  - Tokenization: 8ms
  - Incremental parsing: 12ms
  - Subtree stitching: 3ms
  - Metrics collection: 0.5ms
  - Overhead/margin: 1ms
```

**Fallback Policy**:
- If incremental parse takes >50% of full parse time, switch to full
- If edit affects >30% of tree, consider full parse
- If >100 incremental edits since last full, do full parse (prevent drift)

---

### Memory Budget

**Overhead**:
- Parent links: +8 bytes per node (64-bit pointer)
- Dirty flags: +1 bit per node (packed into flags field)
- Edit history: 100 edits × 48 bytes = 4.8 KB

**For 1k LOC file** (~10k nodes):
- Original tree: ~1 MB
- Parent links: ~80 KB
- Dirty flags: ~1 KB (packed)
- Total: ~1.1 MB (10% overhead)

**For 10k LOC file** (~100k nodes):
- Original tree: ~10 MB
- Parent links: ~800 KB
- Dirty flags: ~12 KB
- Total: ~11 MB (8% overhead)

**Success Criteria**: ≤15% memory overhead for parent links + dirty flags

## VIII. Risk Assessment

### High Risks

**1. Performance Targets Missed**
- **Risk**: Cannot achieve ≤30% of full parse time
- **Probability**: Medium
- **Impact**: High (blocks editor integration)
- **Mitigation**: Early benchmarking, profile hotspots, optimize aggressively
- **Fallback**: Document achieved performance, adjust targets, add fallback to full parse

**2. Correctness Issues**
- **Risk**: Incremental drift or incorrect results
- **Probability**: Medium
- **Impact**: Critical (correctness is non-negotiable)
- **Mitigation**: Extensive testing, property tests, golden test suite
- **Fallback**: Disable incremental parsing if correctness cannot be guaranteed

### Medium Risks

**1. GLR Complexity**
- **Risk**: GLR ambiguity handling too complex
- **Probability**: Medium
- **Impact**: Medium (affects ambiguous grammars only)
- **Mitigation**: Start with deterministic grammars, add GLR later
- **Fallback**: Disable incremental for ambiguous grammars initially

**2. Memory Overhead**
- **Risk**: Parent links + dirty flags exceed budget
- **Probability**: Low
- **Impact**: Medium (affects large files)
- **Mitigation**: Measure early, optimize data structures
- **Fallback**: Make parent links optional (feature flag)
