# Milestone 1: Runtime Integration & Tree-sitter-like API ✅

## Overview

This milestone delivers a minimal rust-sitter-runtime crate with a Tree-sitter-compatible API that provides the foundation for GLR parsing. The runtime serves as a stable interface that users can develop against while the GLR engine integration is completed.

## Deliverables Completed

### 1. Core Runtime Types (✅)

**Location**: `runtime2/src/`

- **`Parser`**: Main entry point with Tree-sitter-compatible methods
  - `new()`: Create parser instance
  - `set_language()`: Configure language for parsing  
  - `parse()`: Parse input with optional old tree
  - `set_timeout()`: Configure parse timeout

- **`Tree`**: Parse tree representation
  - `root_node()`: Get root node
  - `language()`: Get language used
  - `walk()`: Tree traversal support

- **`Node`**: Individual syntax tree nodes
  - Position information (byte ranges, line/column)
  - Symbol and field access
  - Tree navigation methods

- **`Language`**: Language definition container
  - Parse tables (action/goto for GLR)
  - Symbol metadata and names
  - External scanner support (stubbed)

### 2. GLR Engine Adapter (✅)

**Location**: `runtime2/src/engine.rs`

- **`GlrEngine`**: Wrapper around glr-core
  - `from_grammar()`: Create from Grammar IR
  - `parse()`: Main parsing entry point
  - `parse_incremental()`: Incremental parsing support (stubbed)

- **Integration Points**:
  - Uses `rust-sitter-glr-core::ParseTable`
  - Uses `rust-sitter-ir::Grammar`  
  - Adapter pattern for future SPPF→Tree conversion

### 3. Error Handling (✅)

**Location**: `runtime2/src/error.rs`

- Comprehensive error types:
  - `NoLanguage`: Parser has no language set
  - `Timeout`: Parse timeout exceeded
  - `SyntaxError`: Parse failures with location
  - `Other`: Custom error messages for integration

- Location tracking for error reporting

### 4. External Scanner Support (✅)

**Location**: `runtime2/src/external_scanner.rs`

- **Pure Rust trait**: `ExternalScanner`
  - `scan()`: Main scanning method
  - `serialize()`/`deserialize()`: State management
  
- **FFI structs**: For future C scanner compatibility
  - `TSExternalScanner`
  - `TSExternalScannerVTable`

- **Example**: `IndentationScanner` showing Python-like indent tracking

### 5. Build Configuration (✅)

**Location**: `runtime2/Cargo.toml`

```toml
[features]
default = ["glr-core"]
glr-core = ["dep:rust-sitter-glr-core", "dep:rust-sitter-ir"]
incremental = []
external-scanners = []
arenas = ["bumpalo", "typed-arena"]
```

Feature-gated design allows:
- Building without GLR dependency (stub mode)
- Optional performance features (arenas)
- Future incremental parsing

## Architecture Notes

The adapter pattern in `engine.rs` cleanly separates:
- **Runtime API**: Stable Tree-sitter-compatible surface
- **GLR Core**: Parse table generation and LR automaton
- **Grammar IR**: Language definition and rules

This allows the runtime API to remain stable while GLR implementation evolves.
